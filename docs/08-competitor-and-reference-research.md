# 08 竞品与参考研究

## 参考对象

ImgPull 不是传统镜像加速站，而是帮助用户把公开镜像同步到自己私有仓库的工作流平台。可参考以下产品形态：

- 云厂商容器镜像服务控制台。
- Harbor 项目和 Robot Account 管理。
- DevOps SaaS 控制台。
- CI/CD 任务状态页。
- 文档中心和帮助中心。
- 余额、积分、订单类 SaaS 后台。

## 产品差异

传统镜像加速：

- 用户依赖公共代理地址。
- 平台暴露公共镜像代理入口。
- 用户镜像结果不一定归自己私有仓库。

ImgPull：

- 用户配置自己的私有仓库。
- 平台拉取公开镜像后推送到用户仓库。
- 用户最终从自己的 ACR / Harbor / Registry 拉取。
- 结果可追踪、可审计、可结算。

## 关键体验参考

### 任务可观测

需要像 CI/CD 任务一样展示：

- 排队。
- 执行中。
- 成功。
- 失败。
- 日志。
- attempt。
- 错误码。
- 下一步建议。

### 控制台体验

需要像云服务控制台：

- 仪表盘。
- 表格筛选。
- 任务详情。
- 状态 badge。
- 操作日志。
- 系统健康。

### 文档中心

帮助中心应像文档中心，而不是 FAQ 卡片：

- 搜索。
- 分类。
- 文章列表。
- Markdown 正文。
- 相关文章。

### Worker 节点管理

需要像资源池管理：

- 节点状态。
- 并发。
- 权重。
- 心跳。
- 运行任务。
- 失败率。
- 生命周期操作。

## 竞品功能启发

P0 借鉴：

- Registry V2 push 统一底层能力。
- 任务日志和状态可视化。
- 帮助中心文档化。
- Worker 节点生命周期。
- 失败自动返还积分。

P1 规划：

- 云厂商 namespace 自动同步。
- Provider adapter。
- 更复杂批量导入。
- 团队空间。
- API / CLI。

## 需要避免

- 只做一个公共镜像代理站。
- 用户看不到任务阶段和失败原因。
- 失败扣分不返还。
- 把云厂商 OpenAPI 和 Registry 推送凭据混在一起。
- 把多个目标塞进一条任务导致日志和结算混乱。
- 让普通用户管理 Worker 节点。

## 结论

ImgPull 的核心竞争点是：

- 用户自有仓库。
- 任务闭环可追踪。
- 积分冻结和失败返还。
- Digest 校验。
- Worker 节点可运维。
- 轻量批量导入。
- 文档和错误排查能力。
